# 📖 第 7 课：文本分块详解

## 7.1 为什么需要分块？

在上一课中，我们把整段文本原封不动地存入知识库。这在文本较短时没问题，但当你要处理长文档时，有几个问题：

### 问题 1：嵌入模型的输入限制

OpenAI 的 `deepseek-embedding` 模型最大支持 **8191 tokens**。一篇几万字的文档无法一次性生成向量。

### 问题 2：向量不够精确

把一篇很长的文章变成一个 1536 维的向量，相当于把整本书的内容"压缩"成一个点。信息被平均后，检索精度会大大下降。

**类比：**
> 你想找"第三章讲数据库的那段"，但图书管理员只告诉你"这本书大概讲计算机会讲数据库相关的"。分块就像把书拆成章节索引——这样你能精准定位。

### 问题 3：检索粒度问题

即使用向量搜到了某篇文档，返回的结果是整篇文档。用户还要在几千字的文档里手动找答案。分块后，返回的是精确的段落。

## 7.2 三种分块策略

本项目实现了三种分块策略，适用于不同场景：

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **fixed** | 每 N 字切一刀，前后重叠 | 简单可控 | 可能切断句子 | 结构简单的流水账文本 |
| **paragraph** | 按空行（`\n\n`）切分 | 保持语义完整 | 段落长短不一 | 结构清晰的 Markdown 文档 |
| **recursive** | 多层分隔符递归切分 | 兼顾语义和大小 | 实现稍复杂 | 通用场景，推荐默认使用 |

### fixed — 固定字数分块

```
原始文本（1500 字）：
░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░

分块结果（chunkSize=500, overlap=50）：
[======= 块 0: 0-500 =======]
              [======= 块 1: 450-950 =======]
                            [======= 块 2: 900-1400 =======]
                                          [=== 块 3: 1350-1500 ===]

注意：
- 块 0 和块 1 有 50 字的 overlap 区域（450-500）
- overlap 确保关键信息不会刚好被"切断"
- 最后一块可能不满 500 字
```

### paragraph — 段落分块

```
原始文本：
第一段很长很长很长...
（空行）
第二段
（空行）
第三段也很长...

分块结果：
- 如果"第一段" > 1000 字 → 用 fixed 方式继续切
- 否则 → 整段作为一个块
- "第二段" 直接作为一个块
```

### recursive — 递归分块

这是最智能的策略，它的分隔符优先级为：

```
最大分隔符（优先尝试）
    ↓
① 空行（\n\n）           ← 段落边界
② 换行（\n）             ← 行边界
③ 句尾标点（。！？）      ← 完整句子
④ 句中标点（，；、）      ← 短语
    ↓
最小分隔符（最后尝试）
⑤ 固定字数（兜底）       ← 暴力切割
```

**递归逻辑：**
1. 先用 `\n\n` 切，如果某块还是太大
2. 递归对该块用 `\n` 切，如果某块还是太大
3. 递归对该块用 `。！？` 切……以此类推
4. 所有分隔符都用完了还是太大 → 固定字数暴力切

## 7.3 chunkSize 和 overlap 的选择

### chunkSize 选多大？

- **太小（< 200 字）**：语义不完整，检索到的片段可能缺少上下文
- **太大（> 1000 字）**：向量稀释，检索精度下降
- **推荐：500 字** — 大多数句子和短段落能完整保留

### overlap 的作用

```
无 overlap：                   |块1|  |块2|  |块3|
                               可能把关键信息切在边界上

有 overlap（50 字）：          |  块1  |
                                   |  块2  |
                                      |  块3  |
                               边界附近的信息至少在一个完整块中
```

**推荐值：chunkSize 的 10%（500 字 → overlap=50）**

## 7.4 代码实现位置

分块逻辑在 `lib/chunker.ts` 中实现。

在 `lib/rag.ts` 的 `ingest()` 函数中调用：

```typescript
import { chunkText } from "./chunker";

export async function ingest(content: string, strategy?: ChunkStrategy) {
  const chunks = chunkText(content, strategy);
  for (const chunk of chunks) {
    const embedding = await getEmbedding(chunk.content);
    await insertDocument(chunk.content, embedding);
  }
}
```

## 7.5 课后练习

### 练习 1：观察分块效果

在浏览器的"知识入库"区域，粘贴一段 2000+ 字的长文本，点击存入。观察控制台日志：

```
✂️ 正在分块...
   → 共分为 5 个文本块
🔮 正在处理第 1/5 块...
🔮 正在处理第 2/5 块...
...
```

### 练习 2：对比不同策略

准备一段结构化的 Markdown 文档（有多个段落和标题）：

1. 用 `paragraph` 策略入库
2. 问一个跨段落的问题
3. 用 `fixed` 策略再入库同一段文本
4. 问同样的问题
5. 比较两次回答的准确性

### 练习 3：修改分块参数

在 `lib/chunker.ts` 中修改：

- 把 `chunkSize` 从 500 改成 200
- 把 `overlap` 从 50 改成 20
- 存入同一段文本，对比回答质量

### 练习 4：添加新的分隔符

在 recursive 策略中，英文的句号 `.` 和问号 `?` 没有被纳入分隔符。试着添加它们：

```typescript
// 在 separators 数组中增加一级
/(?<=[.?!])/,    // 英文句尾标点
```
