# 📖 第 2 课：代码结构与核心逻辑

## 2.1 代码分层设计

本项目按"职责分离"原则分为三层：

```
┌─────────────────────────────────────┐
│  UI 层 (app/page.tsx)               │
│  用户看到的页面                      │
├─────────────────────────────────────┤
│  API 层 (app/api/*/route.ts)        │
│  接收请求、返回响应                  │
├─────────────────────────────────────┤
│  业务逻辑层 (lib/rag.ts)            │
│  编排 RAG 工作流                     │
├─────────────────────────────────────┤
│  基础设施层 (lib/db.ts,             │
│   lib/embeddings.ts)                │
│  数据库操作 + AI API 调用            │
└─────────────────────────────────────┘
```

> **为什么这样分层？**
> 每一层只关心自己的事情：
> - UI 层只管"怎么显示"
> - API 层只管"接收什么、返回什么"
> - 业务逻辑层只管"先做什么、再做什么"
> - 基础设施层只管"怎么连数据库、怎么调 API"
>
> 这样修改 UI 不影响业务逻辑，换数据库也不影响 API 层。

## 2.2 三个核心文件（lib/ 目录）

### 2.2.1 `lib/db.ts` — 数据库操作

这个文件封装了所有 PostgreSQL 操作，对外暴露三个函数：

```typescript
// 初始化数据库（创建表）
initDB()

// 插入一条文档（文本 + 向量）
insertDocument(content, embedding)

// 搜索相似文档（按向量距离排序）
searchSimilarDocuments(embedding, limit)
```

**关键设计：连接池**

```typescript
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
});
```

> **为什么用连接池而不是每次都创建连接？**
> 每次创建数据库连接要耗费几十毫秒。
> 连接池预先建好一组连接，请求来了直接取用，
> 用完了归还，极大提升性能。

**关键设计：pgvector**

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS documents (
  id SERIAL PRIMARY KEY,
  content TEXT NOT NULL,
  embedding vector(1536)  -- 1536 维向量
);
```

> **vector(1536) 的 1536 是什么意思？**
> DeepSeek 的 deepseek-embedding 模型把一段文本转换成
> 1536 个浮点数组成的向量。所以数据库的向量字段也要是 1536 维。

### 2.2.2 `lib/embeddings.ts` — AI 能力

这个文件封装了两个 AI 能力：

```typescript
// 文本 → 向量
getEmbedding(text)

// 给 AI 提供资料，让它基于资料回答问题
askAIWithContext(question, context)
```

**关键概念：嵌入向量（Embedding）**

嵌入向量就是把"文字"变成"一串数字"。

```
"猫"     → [0.023, -0.456, 0.789, ...]  // 1536 个数字
"狗"     → [0.031, -0.448, 0.776, ...]  // 和猫的向量很接近
"冰箱"   → [0.892, 0.123, -0.567, ...]  // 和猫的向量差很远
```

> **为什么叫"嵌入"？**
> 想象我们把 "猫"、"狗"、"冰箱" 这三个词放到一个 1536 维的空间里。
> "猫"和"狗"在这个空间里距离很近（都是动物），
> "冰箱"离它们很远（不是动物）。
> 这就叫"语义嵌入"——把语义关系嵌入到几何空间中。

**关键概念：System Prompt**

```typescript
const response = await openai.chat.completions.create({
  model: "deepseek-chat",
  messages: [
    {
      role: "system",
      // 这个 system message 告诉 AI 它的身份和规则
      content: "请基于以下资料来回答用户的问题...",
    },
    {
      role: "user",
      content: `=== 参考资料 ===\n${context}\n\n=== 用户问题 ===\n${question}`,
    },
  ],
  temperature: 0.3,  // 低温度 = 更保守的回答
});
```

> **temperature 是什么？**
> 控制 AI 的"创造性"：
> - temperature=0：每次都给出最确定的回答
> - temperature=1：每次回答可能会有不同
> - 对于知识问答，我们想要准确而非创造，所以用 0.3

### 2.2.3 `lib/rag.ts` — RAG 工作流编排

这个文件把前面的功能串起来，对外暴露两个函数：

```typescript
// 初始化
initRAG()

// 知识入库
ingest(content)

// 知识问答
query(question)
```

> **为什么需要这层"编排"？**
> 如果调用者（API 层）直接调用 getEmbedding + insertDocument，
> 代码会变成面条代码。rag.ts 像个"指挥官"，
> 告诉各个模块"你先做、他再做、最后你来做"。
> 这样 API 层只需要一句 `await ingest(content)`。
