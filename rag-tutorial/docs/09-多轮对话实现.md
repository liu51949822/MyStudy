# 📖 第 9 课：多轮对话实现

## 9.1 为什么需要多轮对话？

### 单轮问答的局限

在 rag_2 中，我们的 RAG 是这样的：
```
用户：什么是 RAG？
AI：RAG 是检索增强生成...
用户：它有什么优点？    ← AI 不知道"它"指什么
AI：？？？"它"是什么？
```

因为每次提问都是"独立的"，AI 没有记忆。

### 多轮对话的魔力

```
用户：什么是 RAG？
历史：[用户说"什么是 RAG？"，AI 说"RAG 是..."]
用户：它有什么优点？
历史：[上一条历史 + "RAG 是..." + "它有什么优点？"]
AI：RAG 的优点包括提高准确性、结合私有知识...
     ↑ AI 从历史看到"它"=RAG
```

**这就是多轮对话的核心价值：AI 能理解"上下文中的指代"。**

## 9.3 关键代码解读

### chat-history.ts — 会话管理

```typescript
// 用 Map 存储所有会话（key=sessionId, value=消息数组）
const sessions = new Map<string, ChatSession>();

// 创建新会话
export function createSession(): string {
  const id = crypto.randomUUID();  // 生成唯一 ID
  sessions.set(id, { id, messages: [], createdAt: Date.now() });
  return id;
}

// 获取最近的 N 条消息（默认 6 条 = 3 轮对话）
export function getRecentMessages(sessionId: string, limit = 6) {
  const session = sessions.get(sessionId);
  return session.messages.slice(-limit);
}
```

> **为什么用 crypto.randomUUID()？**
> Node.js 18+ 内置，不需要额外安装 uuid 包。
> 生成的 ID 格式如 "a1b2c3d4-e5f6-7890-abcd-ef1234567890"。

### embeddings.ts — 注入历史到对话

```typescript
export async function askAIWithContext(
  question: string,
  context: string,
  history?: { role: string; content: string }[]
) {
  const messages = [
    { role: "system", content: "你是一个知识库助手..." },
    // 把历史消息插入到这里 ↑
    ...(history || []),
    // 当前问题放最后
    { role: "user", content: `=== 参考资料 ===\n${context}\n\n=== 问题 ===\n${question}` },
  ];

  return await openai.chat.completions.create({ messages, ... });
}
```

> **为什么历史消息插在 system 和 user 之间？**
> 这是 OpenAI 的推荐方式。模型按顺序处理消息：
> 先知道"你是谁"（system），再看"刚才聊了什么"（history），
> 最后看到"现在要回答什么"（user）。

### rag.ts — 带历史的查询

```typescript
export async function query(question: string, sessionId?: string) {
  // 没有 sessionId？创建新会话
  if (!sessionId || !getSession(sessionId)) {
    sessionId = createSession();
  }

  // 取最近 6 条历史（3 轮对话）
  const history = getRecentMessages(sessionId, 6);

  // ...向量搜索...拼接上下文...

  // 带历史问 AI
  const answer = await askAIWithContext(question, context, history);

  // 保存到历史
  addMessage(sessionId, "user", question);
  addMessage(sessionId, "assistant", answer);

  return { answer, sources, sessionId };
}
```

## 9.4 Session 的生命周期

```
用户打开页面
  └→ sessionId = null

用户发送第一条消息
  ├→ 调用 /api/chat { question: "什么是RAG？" }
  ├→ rag.ts 发现 sessionId=null，创建新会话
  ├→ 返回 { answer: "...", sessionId: "abc-123" }
  └→ 前端保存 sessionId = "abc-123"

用户发送第二条消息
  ├→ 调用 /api/chat { question: "它有什么优点？", sessionId: "abc-123" }
  ├→ rag.ts 找到已有会话，加载历史
  ├→ AI 看到历史 → 正确回答
  └→ 前端继续使用同一个 sessionId

用户刷新页面
  └→ sessionId 丢失（存在内存里）
  └→ 发消息时会创建新会话
```

> **为什么刷新页面会话会丢失？**
> sessionId 存在 React 的 useState 里，刷新页面就重置了。
> 生产环境中 sessionId 应该存在 localStorage 或 cookie 中。

## 9.5 历史截断（Token 管理）

```typescript
// chat-history.ts
export function getRecentMessages(sessionId: string, limit = 6) {
  return session.messages.slice(-limit);
}

export function trimHistory(sessionId: string, maxMessages = 20) {
  const session = sessions.get(sessionId);
  if (session && session.messages.length > maxMessages) {
    session.messages = session.messages.slice(-maxMessages);
  }
}
```

**为什么要限制历史长度？**
- GPT 模型有最大 Token 限制（gpt-4o-mini = 128K tokens）
- 但给 AI 太多历史 = 浪费 tokens = 花钱更多
- 通常保留最近 3-5 轮对话就够了
- 太早的历史对当前对话帮助不大

## 9.7 课后练习

1. **"清除对话"按钮**：在 UI 中添加一个按钮，重置 sessionId 和消息列表
2. **会话持久化**：把 sessionId 存到 localStorage，刷新页面不丢失
3. **会话列表**：在聊天框旁边显示所有活跃会话，可以切换
4. **自定义历史长度**：在设置中让用户选择保留几轮对话（3/5/10）
5. **Token 计数器**：显示当前会话已使用的 token 数量（粗略估计）
